It is part of the jdk.httpserver module. The library is intended primarily for development, testing, and simple applications that need to embed an HTTP server without adding a large external dependency. It is not designed for high-performance, production-grade scenarios, where dedicated servers like Netty, Jetty, or embedded Tomcat are typically preferred. . Core Concepts 2.1. HttpServer The HttpServer class is the central component of the library. It is responsible for creating, configuring, and managing the server's lifecycle. An HttpServer instance is created using a factory method, HttpServer.create(). It is configured by associating it with one or more HttpHandler instances via createContext(), which maps specific URI patterns to request handlers. The server is started with the start() method, and can be gracefully stopped with the stop() method. 2.2. HttpHandler The HttpHandler interface defines the contract for processing an HTTP request. It contains a single abstract method, handle(HttpExchange exchange), which is invoked for each incoming request that matches a context defined on the HttpServer. The implementing class (or lambda expression) is responsible for inspecting the request, generating a response, and sending it back to the client. A single HttpServer can have multiple HttpHandlers, each linked to a different URI path or pattern. 2.3. HttpExchange An HttpExchange object represents a single request-response interaction. It serves as the primary conduit of communication between the client and the HttpHandler. The HttpHandler receives an HttpExchange object for each request, which contains all the details of the incoming request (method, headers, path, query parameters, and body) as well as the necessary objects to construct and send the response. The HttpExchange provides methods such as getRequest() to access the incoming request and getResponseBody() to obtain an OutputStream for writing the response body. After the response body is sent, the handler must call sendResponseHeaders() with the appropriate status code and content length. 2.4. Executors and Concurrency By default, the HttpServer uses a cached thread pool to process incoming requests concurrently. This means that each new request is typically handled by a new thread, which can be efficient for a small number of requests but can lead to resource exhaustion (thread explosion) under heavy load. For more control over concurrency, a developer can provide a custom java.util.concurrent.Executor instance to the server using the setExecutor(Executor executor) method. This allows for fine-tuning the thread pool's size, behavior, and lifecycle, enabling the server to manage resources more predictably and efficiently in high-traffic scenarios. HttpServer The `HttpServer` class is the main entry point for managing the server instance. It encapsulates the server's configuration and lifecycle. * **`static HttpServer create(InetSocketAddress addr, int backlog)`**: Factory method to create an `HttpServer` instance bound to a specific network address (`InetSocketAddress`) and a queue size (`backlog`). * **`void setExecutor(Executor executor)`**: Configures the `Executor` service that will handle incoming requests. This is used to control concurrency and thread management. * **`void createContext(String uri, HttpHandler handler)`**: Associates a specified URI pattern (e.g., "/" or "/api/*") with an `HttpHandler`. The handler will be invoked for any request whose path starts with the given URI. * **`void start()`**: Starts the server, making it ready to accept connections. * **`void stop(int delay)`**: Initiates a shutdown of the server. The `delay` parameter specifies the time in seconds to wait for in-flight requests to complete before terminating. * **`int getPort()`**: Returns the port on which the server is listening. * **`Set getContexts()`**: Returns a set of all configured URI contexts. #### 4.2. HttpHandler The `HttpHandler` interface defines the contract for processing HTTP requests. Implementations of this interface contain the application's business logic. * **`void handle(HttpExchange exchange)`**: The core method of the handler. It is called by the server for each incoming request that matches a registered context. The method must inspect the `HttpExchange` object and construct a response, including sending headers and a body. * **`HttpServer attach(String uri)`**: (Extension method) A Java 8+ feature that allows chaining context creation directly from a handler instance. #### 4.3. HttpExchange The `HttpExchange` object represents a single client request and the corresponding server response. It acts as a message container and is passed to the `HttpHandler`. * **`HttpRequest getRequest()`**: Returns an object that provides access to the details of the incoming HTTP request, such as the method, headers, query parameters, and path. * **`HttpResponse getResponse()`**: Returns an object to access the HTTP response being constructed. * **`Headers getResponseHeaders()`**: Provides access to the response headers. The handler can set content types, caching policies, and other metadata here. * **`void sendResponseHeaders(int statusCode, long contentLength)`**: Sends the HTTP status line and response headers to the client.
